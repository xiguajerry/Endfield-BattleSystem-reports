    TryGetData 逻辑¶

    入口行为: AbilityEntityTable::TryGetData

    重要行为:

    • 查找方式是 O(n) 线性扫描，没有内部字典缓存。

    m_mainAssetLoader.Load<T>(hash) 之后¶

    m_mainAssetLoader 不是在 OnPreInit 里创建的，而是在 DataManager::.ctor 里初始化:

    因此 AbilityEntityTable 走的是第二个 loader，RootCategory.Main = 0。 m_initialAssetLoader 则使用 RootCategory.Initial = 1，给 initial config 用。 SimpleAssetLoader 构造函数 会:

    m_handles = new HashSet<FAssetProxyHandle>()
m_channel = channel
m_category = category
m_priority = priority
如果远程配置开启资源堆栈追踪，则保存 StackTrace

    SimpleAssetLoader.Load<T> 的同步加载路径:

    从泛型上下文取得 typeof(T)
ResourceManager.Load(hash, typeof(T), this.m_category, EResourceRequestPriority.Default)
SimpleAssetLoader.HandleAsset(handle)
return FAssetProxyLoaderHandle(handle)

    非泛型版本 SimpleAssetLoader.Load(StringPathHash, Type) 逻辑相同，只是 Type 由调用者传入。 HandleAsset 只负责生命周期管理:

    if (!handle.IsValid())
 return
m_handles.Add(handle)

    也就是说 m_mainAssetLoader 本身不解析 manifest，也不直接读 bundle；它只是把 ResourceManager 返回的 handle 记到自己的 HashSet，方便后续 ClearAll / Dispose 时统一释放。 下层 ResourceManager.Load:

    flowchart TD
    N0["ResourceManager.instance"]
    N1["BundleResourceManager.LoadUntracked(hash, type, category)"]
    N2["new FAssetProxyHandle(untrackedHandle)"]
    N0 --> N1
    N1 --> N2

    ResourceManager.instance 最终取的是 Singleton<BundleResourceManager>.instance。所以 gameplay config 的同步资源加载 实际进入 BundleResourceManager。 BundleResourceManager.LoadUntracked:

    assetProxy = _LoadAssetInternal(hash, type, category, isAsync=false, priority=Default)
if (assetProxy != null)
 return new FAssetProxyUntrackedHandle(assetProxy, ELogChannel.Res)
LogErrorWithLuaStack("加载资源失败 {0}: {1} {2}", category, path, type)
return zero/default FAssetProxyUntrackedHandle

    BundleResourceManager._LoadAssetInternal 才是真正查 manifest 并进入 bundle loader 的地方:

    if (m_manifest != null &&
 m_manifest.TryGetAssetInfoFromPath(hash.hash, out info)) {
 assetProxy = m_loadableMgr.LoadAsset(info, type, isAsync=false, priority=Default)
 if (assetProxy.status == 3 /* FailedToLoad */) {
 LogError("加载资源失败 {0}: {1} errorCode: {2}", category, path, errorCode)

 } else {

 }
}
return null

    其中 RuntimeManifestBinary.TryGetAssetInfoFromPath 使用的是 StringPathHash.hash，也就是前面算出的 1018030359377152006。 BundleLoader.Manager.LoadAsset 继续做 asset proxy 缓存:

    groupID = ++s_operationGroupIDGenerator

if (_assetPools.TryGetValue(key, out asset) == false) {
 asset = BundleProxyContainer.NewAssetProxy(info, type)
 _assetPools.Add(key, asset)
}
如果 asset 已经 `Unloaded(status == 4)`，记录错误日志。

return _LoadAssetProxy(asset, groupID, isAsync=false, priority=Default)

    同步路径的关键结论:

    • AbilityEntityTable.asset 是否存在，先由 DataManager.LoadAsset<T> 的 ResourceManager.CheckExists(hash, RootCategory.Main) 过滤一次。

    • 真正加载时又会在 BundleResourceManager._LoadAssetInternal 里通过 RuntimeManifestBinary.TryGetAssetInfoFromPath(hash.hash, out info) 查 manifest。

    • BundleLoader.Manager 以 (pathHashHead, type) 组成 asset proxy cache key； 同一个资源和类型重复加载时会复用 _assetPools 中的 AssetProxy。

    • SimpleAssetLoader 持有的是 FAssetProxyHandle 集合，不持有实际 asset； 实际对象挂在 AssetProxy 上，最终由 FAssetProxyHandle.Get<T>() 取出。

    AssetProxy / BundleProxy 深层同步加载¶

    AbilityEntityTable 这条链路传给 BundleLoader.Manager.LoadAsset 的 isAsync=false，所以后续走同步分支。资源系统里的状态枚举是:

    BundleProxyContainer.NewAssetProxy(info, type) / AssetProxy.Init 会把 manifest 查出的 AssetInfoPack 复制到 AssetProxy.m_info，并记录:

    m_type = typeof(AbilityEntityTable)
m_isComponentType = ResourceConsts.COMPONENT_TYPE.IsAssignableFrom(m_type)
status = Wait
_objectID = FAssetObjectID.GetNewAssetID()
m_handlesSet = HashSet<int>

    AbilityEntityTable 是 ScriptableObject，不是 Component，所以后面不会走 GameObject.GetComponent(type) 分支。 BundleLoader.Manager._LoadAssetProxy 是 asset proxy 进入加载操作的核心:

    如果同一个资源此前已经在异步加载，_LoadAssetProxy 会检查冲突。配置允许时会 WaitForAllAsyncOperationsToComplete，否则会强制完成当前 asset bundle operation。 但 AbilityEntityTable 在预加载阶段通常是同步加载，重点是 _FastLoadAssetSync。 实际步骤:

    assInfo = asset.GetAssetInfo()
bundle = GetOrCreateBundleProxy(assInfo.bundleData.hashName, cancelRecycle=true)
_LoadBundleProxySync(ref assInfo, bundle, dep=0)
asset.SetBundle(bundle)

    _LoadBundleProxySync 会递归处理 bundle 依赖:

    manifest.TryGetBundleDirectDeps(bundleHashName, deps)
for each depHash:

 _LoadBundleProxySync(ref depInfo, depBundle, dep + 1)

    UnityEngine.AssetBundle::LoadAssetByHash_Internal(System.Int64,System.Type)

    然后执行:

    assetBundle.LoadAssetByHash_Internal(hash.hash, type)

    这里传入的 hash.hash 是 AssetInfoPack.assetData.pathHashHead。对 AbilityEntityTable.asset 来说，这个值对应 manifest 中 asset 路径的 hash； type 是 typeof(AbilityEntityTable)。 AssetProxy._FinishWithAsset 会把 Unity 返回对象写入 AssetProxy.m_asset，并将状态置为 SuccessToLoad。如果 Unity 对象为空或 后续校验失败，则写入错误码并置为 FailedToLoad，上层 BundleResourceManager._LoadAssetInternal 会识别 status == 3，记录加载失败并 Release() 该 proxy。 最后回到 DataManager.LoadAsset<T> 的 handle.Get<T>()。取对象链路是:

    bundles 模块复用评估¶

    bundles 模块已经覆盖了 Unity 资源读取的下半段，适合作为离线还原 AbilityEntityTable 的基础。 可直接复用的部分:

    能力 已有实现 对还原表的作用
    Bundle 解包 BundleFile / WebFile / UnityCompression 读取 UnityFS/UnityWeb/ENCR bundle，解压 block 和内部 stream file
    Endfield VFS chunk 读取 AnimeStudioAssetScanner + EndfieldVfsFileResolver 支持 .chk/.blc 结构、手动 offset、加密 chunk 解密
    Unity object reader SerializedObjectReader 读取 PPtr、AssetBundle、MonoBehaviour、MonoScript 等常见对象
    通用 TypeTree 值树 TypeTreeReader / TypeTreeValueFormatter 可以把未知 Unity 对象按 TypeTree 读成 Map/List
    AssetBundle container AssetBundleObject 可从 container/preload table 找到 asset path 到 object pointer 的关系
    跨文件依赖追踪 AnimeStudioAssetMapBuilder 已有私有逻辑能解析 externals、追踪 PPtr、必要时扫描 raw object pointer
    manifest.hgmmap 解析 HypergryphManifestReader 能把 manifest 转成 assetByHash、bundleByIndex、bundleByHash
    MemoryPack schema MemoryPackSchemaRepository / MemoryPackDumper 可复用为 AbilityEntityTemplateData 字段清单和 JSON 目标结构

    这些能力说明: 还原 AbilityEntityTable 不需要重写 Unity bundle/asset 文件解析。 主要缺口在更上层的“资源定位 + 自定义 ScriptableObject 对象图还原”。 还缺的关键能力:

    1. 可用 manifest 输入和物理 bundle 定位。 运行时从 StringPathHash(1018030359377152006) 查 manifest.hgmmap，得到 AssetInfo.bundleIndex，再找 Bundle.name/hashName。项目已有 HypergryphManifestReader，但当前 AKEDataMining/data/Bundles/manifest.json 是 0 字节；因此本地数据目前只能确认 AbilityEntityTable.asset 的 path hash，不能直接确认对应 bundle 文件。 要真正还原，需要原始 Data/Bundles/Windows/manifest.hgmmap，或已经转换好的非空 manifest.json，以及对应 bundle/chunk 文件。

    结论¶

    AbilityEntityTable 是一个很薄的 ScriptableObject 配表层:

    • 持有 TemplateDataConfig 列表。
    • 用 abilityEntityId == TemplateDataConfig.data.id 查找模板。
    • 输出 AbilityEntityTemplateData，供 SpawnAbilityEntity 执行阶段做生成前校验和读取模板参数。
    • 自身没有缓存、生成、生命周期或网络逻辑。
    • 资源本体来自 Assets/Beyond/DynamicAssets/GameData/GameplayConfig/AbilityEntityTable.asset。

    • 加载时机是 GamePreloadState 预加载阶段的 DataManager.LoadGameConfigs -> _LoadConstConfigs。 其中 AbilityEntityTemplateData 是 EntityTemplateData union tag 0。

    • 实体实际分配时还会通过 DataManager.TryGetEntityTemplate(ObjectType.AbilityEntity, id) 加载或复用 GameData/AbilityEntity/data_{id}.asset 单体模板缓存。 从实现看，风险点主要有三类:

    • AbilityEntityTable.TryGetData 为线性扫描，配置规模增大时会随 SpawnAbilityEntity 执行频率线性增加成本。

    • MemoryPack 解析依赖 57 个字段的顺序和类型精确匹配；字段漂移会造成反序列化失败或错位。
