    CharacterTemplateData 管理链路¶

    结论¶

    CharacterTemplateData 不是一张集中加载的角色表。它是单个角色实体模板的数据体， 由 TemplateDataConfig 这个 Unity ScriptableObject 包一层，再由 DataManager 按 (ObjectType.Character, templateId) 懒加载和缓存。 角色实体创建时的核心链路是:

    flowchart TD
    N0["CharInfo.templateId"]
    N1["DataManager.TryGetEntityTemplate(ObjectType.Character, templateId, out data)"]
    N2["DataManager._TryGetEntityTemplateById(ObjectType.Character, templateId, out data)"]
    N3["ResourceRouter.GetEntityTemplateDataConfigFullPath(Character, templateId)"]
    N4["Assets/Beyond/DynamicAssets/GameData/CharacterData/data_{templateId}.asset"]
    N5["LoadAsset&lt;TemplateDataConfig&gt;(StringPathHash(path))"]
    N6["TemplateDataConfig.data"]
    N7["m_entityIdTemplates[Character][templateId] = data"]
    N8["EntityAllocator.AllocateObject(data, charInstId, ...)"]
    N9["Entity.Construct(data)"]
    N10["CharacterRootComponent.characterTmplData = (CharacterTemplateData)entity.templateData"]
    N0 --> N1
    N1 --> N2
    N2 --> N3
    N3 --> N4
    N4 --> N5
    N5 --> N6
    N6 --> N7
    N7 --> N8
    N8 --> N9
    N9 --> N10

    因此，CharacterTemplateData 的管理者是 DataManager 的实体模板缓存； 具体角色实体只持有对同一份模板对象的引用，不拥有独立拷贝。

    类结构¶

    CharacterTemplateData 自身字段很少:

    objectType getter 的实际汇编是 mov eax, 0x8; ret， 也就是 ObjectType.Character。 真正承载模板内容的是父类 EntityTemplateData:

    再往上是 BaseTemplateData / GameDataWithId，提供 id、name、 factionIndex 等基础字段。 所以一个角色模板的主要内容不是 animConfigPath 和 bodyType，而是 componentList 里的组件数据，例如根组件、移动、模型、动画、能力系统等。

    资源包装¶

    角色模板资源实际加载的是 TemplateDataConfig:

    DataManager._TryGetEntityTemplateById() 加载到 TemplateDataConfig 后， 取 config.data 作为 EntityTemplateData 返回。对于 ObjectType.Character 来说，这个 data 的运行时类型就是 CharacterTemplateData。

    DataManager 内部缓存¶

    DataManager 与实体模板相关的字段:

    DataManager 构造函数会创建这些字典对象。与 CharacterTemplateData 直接相关的是:

    • m_entityIdTemplates: 运行时模板缓存。第一层 key 是 ObjectType， 第二层 key 是 templateId。

    • m_entityTypedTemplates: 无 id 的类型模板缓存，用于一些通用实体模板。 正常角色模板不会走这里。

    • m_entityTemplateDataConfigDict: 字段存在，但在当前分析到的 _TryGetEntityTemplateById() 普通路径中，最终缓存的是 TemplateDataConfig.data 到 m_entityIdTemplates，不是把 TemplateDataConfig 本身作为角色模板主缓存。

    角色模板路径¶

    ResourceRouter.GetEntityTemplateDataConfigFullPath(ObjectType.Character, templateId) 拼出的路径是:

    Assets/Beyond/DynamicAssets/GameData/CharacterData/data_{templateId}.asset

    例如:

    Assets/Beyond/DynamicAssets/GameData/CharacterData/data_chr_0004_pelica.asset

    这和此前 character-template-data-asset-location.md 中定位出的角色资源路径一致。

    实体创建时如何使用模板¶

    EntityAllocator.AllocateObject(data, id, param, levelData) 会从实体池里拿到 Entity，然后调用:

    Entity.Construct(EntityTemplateData data)

    Entity.Construct() 做的关键事包括:

    • 把传入模板保存到 Entity.templateData 槽位。
    • 读取模板的 objectType，写入实体自己的类型字段。
    • 根据模板继续构造实体组件。

    CharacterRootComponent.AssignData(BaseComponentData) 之后会从所属实体取回 templateData，转换成 CharacterTemplateData，写入自己的属性:

    CharacterRootComponent.characterTmplData = (CharacterTemplateData)entity.templateData

    CharacterRootComponent.bodyType getter 也直接从 characterTmplData.bodyType 读取；汇编里可以看到它从 characterTmplData + 0x58 拷贝 BodyTypeDef。 这说明角色根组件不单独管理 body type，它只是引用角色模板里的字段。

    componentList 的按类型缓存¶

    EntityTemplateData.FindComponentData(Type type) 管理 componentList 的查找缓存:

    也就是说，CharacterTemplateData 内的组件数据本身也不是每次线性扫描。 第一次按类型查找后，会在模板对象自己的 m_componentDataCache 里记住结果。 因为角色实体共享同一份模板对象，这个缓存也是模板级别共享的。

    管理语义¶

    整体语义可以概括为:

    • CharacterTemplateData 是不可变配置语义的数据对象，代表一个 templateId 对应的角色实体模板。

    • DataManager 按需加载 TemplateDataConfig asset，并把其中的 EntityTemplateData data 缓存到 m_entityIdTemplates[ObjectType.Character]。

    • 角色实例数据，例如服务端实例 id、等级、战斗状态、generation 等，不在 CharacterTemplateData 内，而在 CharInfo、SCENE_CHARACTER、实体运行时字段和组件状态里。

    • Entity 和 CharacterRootComponent 只持有模板引用，用它初始化组件和读取静态配置。

    • 组件数据的查询由 EntityTemplateData.m_componentDataCache 做模板级缓存。
