    PlayerController 普攻连段与打断缓存¶

    结论¶

    普攻连段没有存在 PlayerController 里。PlayerController 只负责把输入转换成 AttackCommand，以及在按住/续按窗口内继续补发 AttackCommand。 真正决定“下一次普攻是哪一段”的状态在 AbilitySystem.ComboController:

    flowchart TD
    N0["PlayerController.CastNormalAttack"]
    N1["PlayerCommandController.CreateCommand(Attack)"]
    N2["CharacterMainCtrlSystem.OnExecutePlayerCommand(Attack)"]
    N3["AbilitySystem.RefreshNextSkillRequest(Attack)"]
    N4["ComboController.m_nextSkillRequest = { nextSkillId, remainingTime, allowToCastNextSkill }"]
    N5["CenterAttackState._TryCastAttack()"]
    N6["AbilitySystem.GetNextAttackId()"]
    N7["ComboController.nextSkillId"]
    N0 --> N1
    N1 --> N2
    N2 --> N3
    N3 --> N4
    N4 --> N5
    N5 --> N6
    N6 --> N7

    因此冲刺、下落攻击、战技、连携技打断当前普攻时，段数不靠 PlayerController 里的计数器保存，而靠技能时间轴不断注册的 ComboController mapping/allow 窗口继续推进。 这里需要区分短期请求缓存和长期连段状态：NextSkillRequest 不是一个能无条件跨状态 保留的长期变量。CenterAttackState.OnLeave() 会调用 AbilitySystem.ClearSkillRequest()； ComboCacheAction.OnEnd() 也会移除自己的 mapping handle，并且当 cacheEndByAction == true 时，会清掉由该 modifier 产生的请求。也就是说“打断后不重置” 更准确地说是: 技能数据在打断前后仍能提供正确的下一段 mapping/allow 窗口，或者打断动作 自身重新注册了后续 mapping，玩家输入再在 cache time 内刷新/消费它。

    时间限制与允许窗口¶

    ComboController.Tick() 对 m_nextSkillRequest 做两件事:

    ComboController.nextSkillId 只有在请求存在且 allowToCastNextSkill == true 时才返回 真实技能 id；否则返回空字符串:

    _AllowNextSkill(skillId) 会从当前 AbilitySystem 的技能表取出目标 Skill， 然后调用 AbilitySystem.CheckCanInterruptCurSkill(newSkill)。实际“哪一帧开始可以接下一段” 由技能动作窗口控制，常见配置是:

    {
 "_startFrame": 16,
 "_endFrame": 27,
 "$type": "Beyond.Gameplay.Core.AllowNextSkillAction+Data, Gameplay.Beyond",
 "allowedSkillIdList": ["chr_0004_pelica_attack2"]
}

    所以这里有两个窗口:

    • AllowNextSkillAction 写入“当前技能从哪一帧开始允许被这些 skillId 打断”。

    AbilitySystem.CheckCanInterruptCurSkill(newSkill) 会先遍历 ComboController.allowedNextSkillPacks，只要任一 pack 的 skillList 包含 newSkill.skillId 就直接允许打断。否则再走“是否等于当前普通战技/连携技 id”、 skillInterruptPriority 与当前技能 canInterrupt 的通用判断。 请求能否真正施放，要同时满足:

    请求未过期 remainingTime > 0
 && 目标 skillId 仍可用
 && 当前技能允许被该 skillId 打断

 "mappingDataList": [
 {
 "cmdType": "Attack",
 "skillId": "chr_0004_pelica_attack2",
 "cacheEndByAction": true,
 "overrideCacheTime": false
 }
 ]
}

    chr_0004_pelica_attack2.json 再把 Attack 映射到第三段:

    {
 "_startFrame": 0,
 "_endFrame": 28,
 "$type": "Beyond.Gameplay.Core.ComboCacheAction+Data, Gameplay.Beyond",
 "mappingDataList": [
 {
 "cmdType": "Attack",
 "skillId": "chr_0004_pelica_attack3",
 "cacheEndByAction": true,
 "overrideCacheTime": false
 }
 ]
}

    当玩家在这个窗口内按下/续按普攻时，RefreshNextSkillRequest(Attack) 会把当前 m_realCommandMapping[Attack] 快照成 NextSkillRequest.nextSkillId。 这个快照有时间限制，也会在状态/动作结束时被清理；它不是“永久保存的段数”。 战技/连携技也是同一套 mapping 系统。RefreshNextSkillRequest(Skill) 会先读取当前 SkillCommand 的输入 skillId，如果它等于当前普通战技就查 NormalSkill mapping， 如果等于当前连携技就查 ComboSkill mapping。例如 chr_0004_pelica_normal_skill.json 会在自己的窗口里写入:

    {
 "cmdType": "NormalSkill",
 "skillId": "chr_0004_pelica_normal_skill",
 "overrideCacheTime": true,
 "cacheTime": { "value": 0.3 }
}
    • 打断前的 mapping/allow 窗口仍在有效帧内，玩家输入先刷新出下一段请求，然后被状态消费。
    • 打断动作自己的 SkillData 继续注册新的 mapping/allow 窗口，后续输入再生成下一段请求。

    如果这些 request/mapping 都过期或被清掉，才会回到普攻链起点。

    打断后为什么不重置¶

    更精确地看，运行时有三层:

    flowchart TD
    Cache["SkillData.ComboCacheAction"]
    Queue["m_mappingModifiers 优先队列"]
    Effective["m_realCommandMapping<br/>当前实际生效 skillId"]
    Input["玩家输入 Attack"]
    Request["RefreshNextSkillRequest<br/>生成 nextSkillRequest"]
    AllowAction["SkillData.AllowNextSkillAction"]
    Allowed["allowedNextSkillPacks"]
    CanCast{"CheckCanInterruptCurSkill(newSkill)"}
    Cast["释放下一段普攻"]
    Wait["等待后续 allow / 超时清理"]
    Cache --> Queue --> Effective
    Effective --> Input --> Request
    AllowAction --> Allowed
    Request --> CanCast
    Allowed --> CanCast
    CanCast -->|允许| Cast
    CanCast -->|不允许| Wait

    AbilitySystem.ClearSkillRequest() 最终清的是 ComboController.m_nextSkillRequest； 如果请求里带了 SkillCastInputData，会先 Dispose()，然后把 nullable 区域清零。 按 modifier 清请求时也类似:

    所以下一次进入 CenterAttackState 时，_TryCastAttack() 仍然是问 AbilitySystem:

    AbilitySystem.GetNextAttackId(target) 如果没有特殊重置条件，就返回 m_comboController.nextSkillId。而 nextSkillId 只在 allowToCastNextSkill == true 时暴露。 因此“被冲刺/战技/下落攻击打断后继续下一段”的实现基础不是一个持久 combo index，而是 SkillData 在合适帧段注册出的 mapping 链。例如:

    • chr_0004_pelica_attack2.json 在 0..28 帧注册 Attack -> chr_0004_pelica_attack3， 并在 18..28 帧允许 chr_0004_pelica_attack3 打断。

    • chr_0004_pelica_dash_attack.json 自己又在 0..20 帧注册 Attack -> chr_0004_pelica_attack4，并在 10..20 帧允许该技能。

    • chr_0002_endminm_dodge.json 在闪避动作中注册 Attack -> chr_0002_endminm_dash_attack，并在后续帧允许这个 dash attack。 具体某个角色“第二段结束后被哪个动作打断，下一次是否从第三段继续”，要落到该角色 相关技能 JSON 的 ComboCacheAction / AllowNextSkillAction 窗口上判断。底层机制已经明确: mapping/request/allow 都是 action 窗口驱动，action 结束和状态离开都会触发相应清理。 如果这段时间太久，ComboController.Tick() 会把 remainingTime 扣到 0 并清空请求； 这就是“普攻能否连上有时间限制”的实现点。

    目前保留的边界¶

    本轮没有继续模拟物理落地、空中速度、碰撞或具体下落攻击触发条件。这里只确认:

    • OnExecutePlayerCommand(Jump) 会在部分状态下创建 Jump offset。
    • 状态机存在 PlungingAttack -> Attack/Dash/Skill/Free 等转换。
    • 下落攻击结束后能否接上哪一段，仍通过 ComboController 的 mapping/request 机制决定。

    如果要继续确认“下落攻击从空中状态到落地攻击结束的精确物理触发帧”，需要进入 movement/physics 相关函数，这部分应单独分析。
