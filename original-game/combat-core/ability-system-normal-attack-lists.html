    AbilitySystem 普攻技能列表来源¶

    结论¶

    curNormalAttackList 和 curBreakingAttackList 都不是 PlayerController 维护的状态。 对可操作角色来说，它们属于角色身上的 AbilitySystem，初始来源是角色模板 componentList 里的 AbilitySystemData.skillDataBundle:

    flowchart TD
    N0["AbilitySystem.InitSkillDataBundle()"]
    N1["m_skillDataBundle = data.skillDataBundle"]
    N2["AbilitySystem._InitSkills()"]
    N3["curBreakingAttackList = m_skillDataBundle.enabledBreakingNormalAttacks"]
    N4["curNormalAttackList = m_skillDataBundle.normalAttackList"]
    N0 --> N1
    N1 --> N2
    N2 --> N3
    N3 --> N4
    flowchart TD
    Start["AbilitySystem._ApplyModeChange()"]
    Base["重置为 skillDataBundle.normalAttackList"]
    Loop{"还有 realEffectedMode"}
    Get["mode = GetModeData(modeId)"]
    Override{"mode 存在且 overrideNormalAttackList"}
    Apply["curNormalAttackList = mode.normalAttackList"]
    End["完成"]
    Start --> Base --> Loop
    Loop -->|有| Get --> Override
    Override -->|是| Apply --> Loop
    Override -->|否| Loop
    Loop -->|无| End

    本轮在 _ApplyModeChange 的窄范围指令扫描中没有看到 curBreakingAttackList 的模式覆盖逻辑；它基本保持为 SkillDataBundle.enabledBreakingNormalAttacks。 下面主线只讲角色。AbilitySystemForNpc 和 AbilitySystemForInt 是 override InitSkillDataBundle() 的特殊派生类，放在后面作为对照，不能直接套到角色普攻链路上。

    角色链路¶

    角色模板是 CharacterTemplateData，它继承 EntityTemplateData。 真正承载组件配置的是继承来的 componentList:

    角色的技能组件配置就是 componentList 中的 AbilitySystemData:

    因此角色路径可以整理成:

    flowchart TD
    N0["DataManager.TryGetEntityTemplate(ObjectType.Character, charTemplateId, out template)"]
    N1["ResourceRouter.GetEntityTemplateDataConfigFullPath(Character, charTemplateId)"]
    N2["'Assets/Beyond/DynamicAssets/GameData/CharacterData/data_' + charTemplateId + '.asset'"]
    N3["m_mainAssetLoader.Load&lt;T&gt;(hash)"]
    N4["得到 CharacterTemplateData / EntityTemplateData"]
    N5["EntityAllocator.AllocateObject(CharacterTemplateData, ...)"]
    N6["用 template.componentList 创建/初始化组件"]
    N7["AbilitySystem 接收 componentList 中的 AbilitySystemData"]
    N8["AbilitySystem.InitSkillDataBundle()"]
    N9["AbilitySystem._InitSkills()"]
    N0 --> N1
    N1 --> N2
    N2 --> N3
    N3 --> N4
    N4 --> N5
    N5 --> N6
    N6 --> N7
    N7 --> N8
    N8 --> N9

    以 Pelica 为例，StringPathHash.json 中能看到角色模板资源:

    Assets/Beyond/DynamicAssets/GameData/CharacterData/data_chr_0004_pelica.asset

    这份 data_chr_*.asset 才是角色运行时 AbilitySystemData.skillDataBundle 的来源。当前 AKEDataMining/data/Json 展开的主要是 SkillData、BuffData、 InteractiveData 和表配置，没有展开这些角色 CharacterData/*.asset 的正文。 所以 SkillData/chr_0004_pelica_attack1.json 能看具体技能逻辑，不能反推 curNormalAttackList 的完整运行时列表。 data_chr_*.asset 的实例 id、模板 id 和资源路径定位链路见 角色 data_chr_*.asset 定位链路。 TableCfg/CharacterTable.json 和 TableCfg/CharGrowthTable.json 也要分开看:

    • CharacterTable 是角色元信息，能看到如 dontInterruptCombo，但没有 skillDataBundle。

    • CharGrowthTable.skillGroupMap 是展示/养成分组。例如 Pelica 的 NormalAttack 分组列出 attack1 到 attack4、power_attack、 plunging_attack_end。它能帮助定位技能 id，但不是 curNormalAttackList 的直接来源。

    • 运行时普攻链的直接来源仍然是角色模板 AbilitySystemData.skillDataBundle.normalAttackList。

    AbilitySystemForNpc.InitSkillDataBundle() 不直接使用 AbilitySystemData.skillDataBundle，而是从 AbilitySystemForNpcData.NpcData 构造一个新的 SkillDataBundle:

    AbilitySystemForInt.InitSkillDataBundle() 也是构造临时 bundle，但来源是交互物属性黑板:

    这两个 override 目前只补 active/passive skill 列表，没有填 normalAttackList 或 enabledBreakingNormalAttacks。因此 Npc/交互物如果没有额外热更补丁， 不应依赖这两个 cur*AttackList 做普通角色式的普攻连段。 角色不用走这两个 override。角色走的是 base AbilitySystem.InitSkillDataBundle()， 也就是直接读取角色模板组件上的 AbilitySystemData.skillDataBundle。
