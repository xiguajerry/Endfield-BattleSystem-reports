    核心数据¶

    Squad 维护当前 leader 的 slot 状态:

    字段 偏移 含义
    leaderIndex 0x10 当前 leader 所在 slot
    leaderId 0x18 当前 leader 的角色实例 id
    slots 0x20 squad slot 列表
    Slot.charInstId 0x18 slot 绑定的角色实例 id
    Slot.character 0x28 slot 当前实体

    PlayerController.mainCharacter 是最终运行时主控引用；Squad.leaderIndex/leaderId 是小队层的 leader 状态。二者通过 _SetMainCharacter 连接。

    SetLeaderByClient 与 Squad.SetLeader¶

    SetLeaderByClient 是外部请求入口，Squad.SetLeader 是真正改 squad leader 状态的入口。

    SetLeaderByClient 不直接做成员归属判断，它只做非空、存活和 squad 状态校验； 具体是否是当前 squad 成员由 Squad.SetLeader 的 slot 匹配决定。

    Squad.Apply 与初始化 leader¶

    _InitLeader 的分叉很关键: 当前 leader 仍有效时直接 _SetMainCharacter； 只有当前 leader 无效并找到 fallback 时，才通过 SetLeader 同步 leaderIndex/leaderId。

    _PostProcessMembers¶

    PlayerController.SetMainCharacter 已经通知 ability/animator；_PostProcessMembers 更偏向 squad member 的 controller、actor data、AI/focus 刷新。

    角色释放与死亡 fallback¶

    释放当前 leader 和当前 leader 死亡最终都会回到 SetLeader， 所以它们会同步更新 leaderIndex/leaderId，再进入 _SetMainCharacter。

    对条件 action 的影响¶

    CheckMainCharacterCondition 不关心 leaderIndex、leaderId、slot 或 squad member 列表。 它只读最终落在 PlayerController.mainCharacter 上的实体引用。 因此实际语义是:

    • 切 leader 成功后，条件立即跟随新主控。
    • 角色重建时，如果重建的是 leader，条件会跟随新实体引用。
    • 当前 leader 释放或死亡后，如果 fallback 成功，条件会跟随 fallback leader。
    • 如果 _SetMainCharacter 因为 newLeader == oldMainChar no-op，则条件结果不会变化， 同时 leader-change 全局事件也不会重复发。
