    战斗控制系统主线¶

    当前结论¶

    角色战斗控制不是单个类完成的，而是四层协作:

    flowchart TD
    N0["CharacterTemplateData"]
    N1["AbilitySystemData.skillDataBundle / modeConfig / dashBuff / plungingAttackData"]
    N2["AbilitySystem 初始化技能实例、当前技能槽、普攻列表、连携条件"]
    N3["PlayerController.PlayerCommandController 缓存玩家输入命令"]
    N4["CharacterMainCtrlSystem 状态机消费命令并切换 Free / Attack / Skill / Dash / PlungingAttack"]
    N5["AbilitySystem.TryCastSkill / TryCastNormalAttack / ComboController 选择并释放具体技能"]
    N0 --> N1
    N1 --> N2
    N2 --> N3
    N3 --> N4
    N4 --> N5

    PlayerController 是本地输入和主控引用的聚合点；真正的战斗状态机在 CharacterMainCtrlSystem，真正的技能集合和命令到技能映射在角色实体的 AbilitySystem。 OnExecutePlayerCommand、Dash enter/leave、Attack enter/update/leave 的非物理主线已经补到可模拟粒度。 以下伪代码来自 类型信息、新提取的

    角色横向差异¶

    大多数角色使用 common_character_perfect_dodge，例外是:

    chr_0030_zhuangfy -> chr_0030_zhuangfy_perfect_dodge

    部分角色有模式覆盖普攻或命令映射:

    角色 mode 覆盖内容
    chr_0016_laevat ult Attack -> chr_0016_laevat_ult_attack1，普攻列表切到 ult attack
    chr_0017_yvonne ult Attack -> chr_0017_yvonne_ult_attack1_1，普攻列表切到 ult attack
    chr_0017_yvonne ult_end Attack -> chr_0017_yvonne_ult_attack3_2
    chr_0017_yvonne talent_1 Attack -> chr_0017_yvonne_attack5
    chr_0027_tangtang ult / ult_end ult 期间改写 Attack
    chr_0030_zhuangfy UltMode Attack -> chr_0030_zhuangfy_attack1_ult，普攻列表切到 ult attack

    这说明 modeConfig 是运行时改变普攻入口的正式机制。AbilitySystem._ApplyModeChange() 会把当前命令映射和 curNormalAttackList 从模板基础值切换到模式值。 下落攻击时长也在模板中:

    角色 startDuration endDuration
    多数角色 0.633 1.267、2.0 或角色特化值
    chr_0030_zhuangfy 0.73 1.267
    chr_0012_avywen / chr_0021_whiten 0.633 4.333

    这些时间属于技能控制流程的配置输入；具体空中/落地判定仍要由移动物理系统驱动。

    主控战斗状态机¶

    CharacterMainCtrlSystem 实现 PlayerController.IPlayerCommandBlocker，因此它既能 阻止命令，也能消费命令，还能在命令执行时驱动主控状态机。 状态:

    黑板:

    状态机主线可以先粗略看成:

    stateDiagram-v2
    [*] --> Free
    Free --> Attack: Attack command
    Free --> Skill: Skill command
    Free --> Dash: Dash command
    Free --> PlungingAttack: in-air attack
    Attack --> Free: attack finish
    Attack --> Attack: attack request allowed
    Attack --> Dash: dash request allowed
    Attack --> Skill: skill request allowed
    Skill --> Free: skill finish
    Dash --> Free: dash finish
    Dash --> Attack: attack allowed after dash
    PlungingAttack --> Attack: landing/end attack allowed
    PlungingAttack --> Skill: skill request allowed
    PlungingAttack --> Free: finish

    这和之前普攻打断结论吻合: 下落攻击、冲刺、战技、连携技并不重置 ComboController 的映射/request。它们只是状态机临时切到其他状态；结束后如果 ComboController 的 NextSkillRequest 还在有效时间内，下一次 Attack 仍能接到 打断前注册的下一段。

    _CheckCenterCanCastNormalAttack():

    这里不是直接判断玩家是否按下攻击键，而是判断 AbilitySystem 里是否已有 下一次攻击请求。这个请求由 PlayerController.CastNormalAttack() / AbilitySystem.RefreshNextSkillRequest(Attack) 建立。

    Attack / Skill 转 Dash¶

    所以战技期间能否冲刺打断，至少受三个门限制:

    • Dash 命令仍然有效。

    CheckConsumePlayerCommand 分支¶

    CharacterMainCtrlSystem.CheckConsumePlayerCommand(cmd) 本体是状态机虚分发包装:

    • CenterStateBase: 默认返回 false。
    • CenterDeadState: 返回 true，死亡状态下所有命令直接 consume。
    • CenterDummyState: 返回 true，无主控/dummy 状态下所有命令直接 consume。
    • CenterDashState: 只有 Attack 且 dashBlockAttackTimer.remainingTime > 0.00001f 时返回 true。
    • CenterFreeState: MoveAxis 返回 false；InImmobilized(7) 命中时返回 true；InDisarmed(6) 或 CantCastAnySkill(4) 命中时只 consume Attack；非 Dash 命令回落到 PlayerController.blockPlayerInput；Dash 命令检查 perfectDodgeBlockDashTimer。

    • CenterSkillState: 只处理 Dash；perfectDodgeBlockDashTimer > 0.00001f 时 consume；否则在当前技能特定状态下检查 curSkill.canDash，不可 dash 时 consume。

    • CenterSkillState.CheckConsumePlayerCommand。

    • 原生只在 cmdType == Dash 时继续判断；其他命令返回 false。
    • perfectDodgeBlockDashTimer.remainingTime > 0.00001f 时直接返回 true。
    • 否则仅当当前 curSkill.skillSpec == 7 时读取 Skill.canDash，返回 !canDash；结合当前 enum 顺序，7 对应 CharacterDashAttack。
    • Skill.get_canDash，逻辑是 Skill.canInterrupt || m_markCanDash。
    • MarkCanDash.ExecuteInternal 会取当前 action environment 的 Ability.curSkill 并调用 Skill.MarkCanDash()，后者 只把 m_markCanDash=true。

    _PlungingAttackToFree() 依赖移动组件状态，暂不做完全模拟。已经确认的非物理部分:

    • InImmobilized 或 InDisarmed 会直接允许退出。

    后台角色技能释放链路¶

    2026-06-01 补充：原游戏把前台主控输入和后台角色技能释放拆成两条链路。 PlayerCommandConfig.CmdType 只有 Attack / Skill / Dash / Jump 等粗粒度输入。 主控角色释放技能时，PlayerController.CastSkill(source, skillId, target, ...) 会创建 CmdType.Skill 的 SkillCommand，并通过 PlayerCommandController.Execute(CmdType.Skill) 进入 CharacterMainCtrlSystem 的 CenterSkillState。 后台角色释放技能时，source != mainCharacter.abilityCom。此分支不会调用 SetLeaderByClient，不会写 PlayerController.mainCharacter，也不会进入主控 center state。 它会：

    flowchart TD
    N0["PlayerController.CastSkill(backgroundAbility, skillId, target)"]
    N1["SkillCommand.AssignData(backgroundAbility, inputView)"]
    N2["backgroundAbility.RefreshNextSkillRequest(CmdType.Skill)"]
    N3["CharacterSkillSystem.CastComboSkill(backgroundEntity, inputView)"]
    N4["CharacterAIBrain.SwitchAIMode(CastSkill)"]
    N5["CharacterCastSkillBehavior._TryCastSkill()"]
    N6["backgroundAbility.TryCastSkill(...)"]
    N0 --> N1
    N1 --> N2
    N2 --> N3
    N3 --> N4
    N4 --> N5
    N5 --> N6

    CharacterSkillSystem._TickToCastSkill() 还会扫描 squad members，对非主控角色检查 HasNextSkillRequest()、CheckCanInterruptCurSkill(nextSkill) 和 CanCastSkill(...)， 满足条件后调用 CastComboSkill(...)。这说明后台角色技能由角色自身 AI/SkillSystem 推进，而不是由主控状态机推进。 连携技 UI 链路为：

    flowchart TD
    N0["ComboSkillPanel.CastComboSkill(charIndex)"]
    N1["BattleManager.CastPendingComboSkill(charIndex)"]
    N2["BattleManager._TryCastComboSkill(slotAbility, comboCastInfo)"]
    N3["AbilitySystem.TryCastComboSkill(...)"]
    N0 --> N1
    N1 --> N2
    N2 --> N3
    • NormalSkill / ComboSkill / UltimateSkill / explicit Skill：允许非主控角色后台释放，不切主控，不改变 CenterState。
    • Attack / PlungingAttack / BreakingAttack：仍是前台限定动作，必须走主控 center state；对非主控 slot 触发时会按现有前台动作语义切主控后执行。
    • BreakingAttack 额外要求目标可选且处于失衡状态。
